# 人机对战（PVE）设计

## 1. 架构

### 组件
在敌方塔（或独立 GameObject）上挂载 `AICommander.cs`，负责自动部署逻辑。

### 与现有系统关系
| 系统 | 关系 |
|------|------|
| `DeploySystem` | AI 共用同一套费用管理？还是独立费用？见下方 1.1 讨论 |
| `DeployUI` | AI 不经过 UI，直接调用部署逻辑 |
| `GridManager` | AI 查询空闲格子、预留格子 |
| `GameManager` | 监听 OnBeat、OnStateChanged 决策时机 |
| `UnitBase` | AI 部署时设置 `isEnemy = true`，方向朝左 |

### 1.1 费用方案（待定）

**方案 A：共用玩家费用**
- 简单，但 AI 和玩家抢费，不好玩
- AI 费用 = 玩家费用 → AI 强则玩家没钱，AI 弱则玩家碾压

**方案 B：AI 独立费用**
- `AICommander` 维护自己的 `_energy`，独立回费
- 回费序列可以不一样（AI 可能更激进或更保守）
- 推荐此方案

**方案 C：按时间波次**
- AI 不花"费用"，而是每隔 N 小节自动出一波兵
- 类似塔防的出怪逻辑
- 最简单，但缺少策略博弈感

> **推荐方案 B**，与玩家对称但有差异空间。

## 2. 部署逻辑

### 2.1 决策时机
- `GameState.Deploy`（备战阶段）：开局时 AI 用初始费用先下一波
- `GameState.Battle`：每小节拍 1（`OnBeat(bar, 1)`）做一次部署决策
- 玩家点击"开战"后，`OnStateChanged(Battle)` 时也触发一次

### 2.2 核心策略

#### 2.2.1 尽量花光费用
- 每次决策时，**只要费用 >= 场上最便宜单位的费用**，就尝试部署
- 尽可能在回费前把费用压到 < 最便宜单位费用

#### 2.2.2 兵种站位
- **近战（attackRange = 1）** → 部署在靠前位置（靠左，接近战场中线）
- **远程（attackRange > 1）** → 部署在靠后位置（靠右，靠近己方塔）
- 原因：近战需要走到前面扛线，远程在后面输出

#### 2.2.3 兵种多样性
- 记录最近 N 次部署的单位类型
- 优先选择与最近 2-3 次部署不同的种类
- 如果某种类连续未使用，提高其权重

#### 2.2.4 费用规划
- 如果当前费用 < 最便宜单位费用 → 跳过
- 如果当前费用刚好够一个高费单位 → 不是无脑下高费，先看是否需要补低费
- 简单策略：剩余费用 > 10 时优先下两个中费而非一个高费，保证场面人数

#### 2.2.5 先锋处理
- 如果 AI 有先锋单位（`isVanguard`），可以部署在队伍中段
- 先锋的作用是冲阵，不需要放在最前面

### 2.3 部署范围（敌方视角）
- 敌方部署范围 = 右半场（cols `Cols/2` ~ `Cols-1`）
- 最左可部署列 = `max(player 最前单位 - 1, Cols/2)`（不超过己方最前单位）
- 最右可部署列 = `cols - 1`（己方塔位置）
- 具体逻辑可以和 `DeploySystem.GetMaxDeployCol()` 对称

### 2.4 部署流程（单次决策）

```
AI决策一次：

1. 收集信息
   - 当前费用 energy
   - 场上己方单位列表（位置 + 类型）
   - 场上敌方单位列表（位置 + 类型）
   - 可用卡牌列表（AIDeckEntry[]）

2. 筛选可用单位（费用 ≤ energy）
   如果没有 → 结束

3. 按优先级排序
   - 最近未使用 > 兵种多样性 > 费用适配

4. 选择部署位置
   - 近战 → 靠前（偏左）
   - 远程 → 靠后（偏右）
   - 调用 CanPlaceUnit 校验

5. 执行部署
   - Reserve 格子
   - 入队 PendingDeploy
   - 扣费
   - 下一拍 OnBeat(beat=1) 时执行
```

## 3. 卡组配置

```csharp
[System.Serializable]
public class AIDeckEntry
{
    public UnitData unitData;
    public int weight = 1;          // 权重，越高越优先选
    public bool isMelee;            // 是否为近战（自动判断或手动指定）
}
```

AI 在 Inspector 里配置一套独立的卡组，和玩家的卡组无关。

### AICommander 接口设计

```csharp
public class AICommander : MonoBehaviour
{
    [SerializeField] private AIDeckEntry[] _deck;   // AI 可用单位
    [SerializeField] private int _startEnergy = 5;  // AI 初始费用
    [SerializeField] private int _barsPerRegen = 15; // AI 回费周期
    [SerializeField] private int _maxEnergy = 8;     // AI 费用上限

    private int _energy;
    private int _barsSinceLastRegen;
    private int _reservedCol;   // 预留列（AI 单线程部署，每次只考虑一列）
}
```

### 与现有系统的交互

1. **部署**：AI 调用 `GridManager.Instance.Reserve/Occupy` + `GameManager.Instance.RegisterUnit`（或借用 `DeploySystem.ExecutePendingDeploys` 风格的逻辑）
2. **费用**：AI 自己管理 `_energy`，不经过 `DeploySystem`
3. **执行时机**：`OnBeat(bar, 1)` 时部署，随后 `ExecutePendingDeploys` 生效
4. **生成单位**：`Instantiate(prefab)` → `SetField(unit, "isEnemy", true)` → `SetField(unit, "_gridPos", ...)` → `RegisterUnit`

## 6. 后续可优化的策略

### 6.1 针对玩家阵容反制
- 检测玩家场上单位类型占比
- 如果玩家近战多 → AI 多出远程
- 如果玩家远程多 → AI 多出近战冲锋

### 6.2 经济运营
- 存费策略：攒到某个费用再下大单位，而不是每回合花光
- 根据局势决定激进度：己方劣势时积极出怪，优势时稳扎稳打

### 6.3 协同推进
- 当 AI 场上人数 > N 时，集中部署在同一侧形成推进
- 当 AI 场上人数 < 3 时，分散部署防守

### 6.4 先锋反制
- 检测到玩家有先锋单位时，在前线部署高血量单位作为"肉盾"
- 避免先锋直接冲脸后排

### 6.5 位置微调
- 不在同一列连续部署（除非该单位占多格）
- 避免单位堆叠在同一列被 AOE 一锅端
